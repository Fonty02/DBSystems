\subsubsection{Redundancies and Generalization}
\begin{itemize}
    \item \textbf{Redundancies:} The number of contracts handled by a team is not stored in the schema, as it can be calculated by counting the number of contracts related to the facility managed by the team. The score of the team can be derived by the following formula: $score = \frac{energy\_efficiency + uptime + AVG(customer\_score)}{3}$ and is not stored in the schema to avoid redundancy. Due to the previous analysis, the redundancy of the efficiency score in the facility is kept, while the redundancy of the sumEnergyOutput in the facility isn't introduced.
    \item \textbf{Generalization:} The schema doesn't contain any generalization since there aren't different operations for theme. There is a \textbf{type} attribute in the \textbf{Customer} and \textbf{Contract} entities so that the business rule related to the type of contract can be enforced.
\end{itemize}
